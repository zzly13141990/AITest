# 软件需求与系统设计文档 - SQL多语句查询与结果管理

## 1 项目概述

### 1.1 项目背景

db-query-java 是一个数据库查询工具，当前版本仅支持单条SQL语句执行。随着用户业务场景的复杂化，需要批量执行多条SQL语句的需求日益增长。本需求旨在增强SQL编辑器的多语句支持能力，提升查询效率和结果管理体验。

### 1.2 建设目标

- 支持多条SQL语句同时查询（按分号分隔）
- 实现查询结果多页签展示，便于结果对比和管理
- 提供结果集二次检索过滤功能
- 优化导出按钮位置和交互方式
- 实现行选中与批量导出功能

### 1.3 适用范围

本需求适用于 db-query-java 项目的 SQL 编辑器页面，包括：

- SQL 编辑区域
- 查询结果展示区域
- 结果导出功能

### 1.4 读者对象

- 前端开发人员
- UI/UX 设计师
- 测试工程师
- 产品经理

## 2 业务需求概述

### 2.1 业务场景

#### 2.1.1 批量数据处理场景

数据库管理员需要执行多条相关的SQL语句（如批量更新、数据迁移等），希望一次性提交并查看每条语句的执行结果。

#### 2.1.2 多结果对比分析场景

数据分析人员需要同时执行多条查询语句，在不同结果页签之间切换对比分析数据。

#### 2.1.3 结果快速定位场景

查询结果集较大时，用户需要快速检索特定数据，而不需要重新执行查询。

#### 2.1.4 选择性导出场景

用户需要从查询结果中选择部分行进行导出，而不是导出全部数据。

### 2.2 用户角色与职责

| 角色 | 职责 |
|-----|-----|
| 数据库管理员 | 编写和执行多条SQL语句，管理查询结果 |
| 数据分析人员 | 对比分析多个查询结果，检索和导出数据 |
| 前端开发 | 实现SQL解析、多页签管理、行选中和导出功能 |
| 测试工程师 | 验证多语句执行、结果过滤、导出功能的正确性 |

### 2.3 业务核心痛点

1. **单语句限制**：当前一次只能执行一条SQL，批量操作效率低下
2. **结果管理不便**：多条语句执行结果无法同时查看和对比
3. **数据定位困难**：大结果集无法快速检索特定数据
4. **导出不够灵活**：只能导出全部数据，无法选择部分行

### 2.4 整体业务流程

```
用户编写多条SQL语句
    ↓
点击执行按钮
    ↓
智能解析SQL（按分号分隔，排除字符串内分号）
    ↓
逐条执行SQL语句
    ↓
创建多个结果页签展示
    ↓
用户可切换页签查看结果
    ↓
支持结果检索过滤
    ↓
支持行选中和批量导出
```

### 3.3 单个功能详细描述

#### 3.3.1 SQL多语句编辑与解析

**功能描述**：支持在单一编辑器中编写多条SQL语句，智能识别语句边界。

**输入**：
- 用户输入的SQL文本（可能包含多条语句，按分号分隔）

**输出**：
- 解析后的SQL语句数组
- 每条语句的执行结果

**业务规则**：
- SQL语句必须以分号作为结束标记
- 智能识别字符串内的分号（不应作为分隔符）
- 多行语句应被正确识别为单条语句
- 支持SELECT、UPDATE、MERGE、DELETE、TRUNCATE、CREATE等所有SQL语句类型

**异常处理**：
- 最后一条语句末尾没有分号时，仍应视为有效语句
- 空语句（连续分号）应被忽略

#### 3.3.2 结果多页签展示

**功能描述**：执行多条SQL后，在结果区域创建多个页签，每个页签对应一条SQL的执行结果。

**输入**：
- 多条SQL语句的执行结果数组

**输出**：
- 多个结果页签，每页签显示对应语句的执行结果

**业务规则**：
- 页签标题显示SQL语句摘要（前20-30个字符）
- 页签按执行顺序排列
- 支持页签切换（点击切换）
- 支持页签关闭（点击关闭按钮或右键菜单）
- 关闭所有页签后显示空状态

#### 3.3.3 结果二次检索过滤

**功能描述**：在每个结果页签内提供搜索框，支持实时过滤结果集。

**输入**：
- 用户在搜索框中输入的关键词

**输出**：
- 过滤后的结果表格（仅显示匹配的行）

**业务规则**：
- 搜索匹配所有列的内容
- 实时过滤（输入即搜索）
- 不区分大小写
- 搜索框清空后恢复显示全部数据
- 过滤结果显示匹配行数统计

#### 3.3.4 行选中功能

**功能描述**：支持点击行号选中单行或多行，用于批量操作。

**输入**：
- 点击行号（普通点击或SHIFT+点击）

**输出**：
- 选中行的高亮显示
- 选中行数据的记录

**业务规则**：
- 点击行号选中单行
- 按住SHIFT点击行号：选中从上次选中行到当前行的范围内所有行
- 未按住SHIFT点击：取消上一个选中，选中当前行
- 选中行有明显的视觉标识（如背景色变化）

#### 3.3.5 导出按钮优化与批量导出

**功能描述**：优化导出按钮位置和样式，支持导出选中行或全部数据。

**输入**：
- 用户点击导出图标按钮
- 当前选中的行数据（如有）

**输出**：
- Excel文件下载

**业务规则**：
- 导出按钮移至分页区域（通常在表格底部）
- 使用图标形式展示（如下载图标）
- 悬停显示提示文字（如"导出Excel"）
- 有选中行时导出选中记录
- 无选中行时导出全部数据

### 3.4 页面/交互简要说明

#### 3.4.1 SQL编辑器区域

- 保持单一编辑器，不拆分为多个页签
- 支持语法高亮
- 支持多行输入
- 状态栏显示当前语句数量

#### 3.4.2 结果页签区域

- 页签位于结果区域顶部
- 页签标题：SQL语句前20-30字符，过长时显示省略号
- 页签右侧显示关闭按钮（×）
- 切换页签时平滑过渡

#### 3.4.3 搜索过滤框

- 位于结果表格上方
- 左侧显示搜索图标
- 支持实时搜索
- 右侧显示匹配行数（如"共100条，匹配10条"）

#### 3.4.4 行号区域

- 行号列位于表格最左侧
- 鼠标悬停时显示手型光标
- 选中行高亮显示
- 范围选中时显示连续高亮

#### 3.4.5 导出按钮

- 位置：分页区域右侧
- 样式：图标按钮（下载图标）
- 悬停效果：显示tooltip提示"导出Excel"
- 点击触发导出

## 4 非功能需求

### 4.1 性能需求

- SQL解析时间 < 100ms（100条语句以内）
- 结果过滤响应时间 < 50ms
- 页签切换响应时间 < 30ms
- 导出大结果集（10万行）时不应阻塞UI

### 4.2 安全需求

- SQL语句应进行参数化处理，防止SQL注入
- 导出的Excel文件不应包含敏感信息（如密码）
- 文件导出时应进行适当的权限检查

### 4.3 兼容性需求

- 浏览器支持：Chrome、Firefox、Safari、Edge（最新两个版本）
- 支持各种数据库类型的SQL语法（MySQL、PostgreSQL、Oracle等）
- 导出的Excel文件支持Office 2016+和WPS

### 4.4 可扩展与可维护性

- SQL解析逻辑应模块化，便于扩展支持更多SQL特性
- 页签管理逻辑应独立，便于后续添加更多页签功能
- 导出功能应支持扩展其他格式（如CSV、JSON）

### 4.5 部署与运维需求

- 无需后端改动（纯前端功能）
- 不影响现有功能的部署和运行
- 前端构建后自动包含新功能

## 5 数据与数据库设计

### 5.1 核心数据实体

本功能主要涉及前端数据结构，不涉及后端数据库变更。

### 5.2 前端数据结构设计

#### 5.2.1 SQL语句解析结果

```typescript
interface ParsedSql {
  id: string;           // 语句唯一标识
  sql: string;          // 原始SQL语句
  summary: string;      // 语句摘要（用于页签标题）
  type: SqlType;        // 语句类型：SELECT/UPDATE/MERGE/DELETE/TRUNCATE/CREATE等
}

type SqlType = 'SELECT' | 'UPDATE' | 'MERGE' | 'DELETE' | 'TRUNCATE' | 'CREATE' | 'INSERT' | 'OTHER';
```

#### 5.2.2 查询结果页签

```typescript
interface ResultTab {
  id: string;               // 页签唯一标识
  sqlId: string;            // 关联的SQL语句ID
  title: string;            // 页签标题
  data: Array<Record<string, any>>;  // 查询结果数据
  columns: Column[];        // 列信息
  selectedRows: number[];   // 选中的行索引
  filterText: string;       // 当前过滤文本
  isActive: boolean;        // 是否为当前激活页签
}

interface Column {
  name: string;             // 列名
  type: string;             // 数据类型
}
```

### 5.3 数据流转关系

```
用户输入SQL
    ↓
SQL解析（识别分号，排除字符串内分号）
    ↓
生成ParsedSql数组
    ↓
逐条执行SQL
    ↓
生成ResultTab数组
    ↓
渲染多页签结果
    ↓
用户交互（切换页签、过滤、选中行、导出）
```

## 6 接口设计

本功能主要为前端功能，不涉及新增后端接口。但需要确保与现有查询接口兼容：

- **现有接口**：`POST /api/query/execute`
- **功能**：执行SQL语句并返回结果
- **调整**：支持接收多条SQL语句数组，返回多条结果数组

**请求体格式**：
```json
{
  "sqls": ["SELECT * FROM table1;", "UPDATE table2 SET ...;"]
}
```

**响应体格式**：
```json
{
  "results": [
    {
      "success": true,
      "data": [...],
      "columns": [...],
      "rowCount": 100
    },
    {
      "success": false,
      "error": "SQL执行错误",
      "errorCode": "42000"
    }
  ]
}
```

## 7 约束与边界说明

### 7.1 技术约束

- SQL语句必须以分号作为语句结束标记
- 智能识别字符串内的分号（不应作为分隔符）
- 多行语句应被正确识别为单条语句
- 最后一条语句末尾可以没有分号

### 7.2 设计约束

- SQL编辑器保持单一，不拆分为多个页签
- 页签数量不应过多（建议限制最大10个，超过时提示用户）
- 导出文件大小应限制（如最大50MB或10万行）

### 7.3 兼容性边界

- 对于不支持的SQL语句类型，应正常执行并返回结果
- 大结果集（如10万行以上）应考虑分页加载
- 特殊字符（如换行符、特殊符号）应正确处理

### 7.4 异常边界

- 部分语句执行失败时，应继续执行后续语句
- 失败语句的页签应显示错误信息
- 空SQL语句应被忽略，不创建页签

## 8 待确认问题与后续迭代规划

### 8.1 待确认问题

1. SQL语句类型检测的准确性要求（是否需要精确识别所有DDL/DML类型）
2. 页签数量上限（当前建议最大10个）
3. 导出文件格式是否需要支持CSV等其他格式
4. 是否需要支持批量执行的事务控制（BEGIN/COMMIT）
