## 二、Critical 问题

### C1. Schema 精简策略与 Token 预算矛盾

| 字段 | 内容 |
|------|------|
| **位置** | §2.1 FR-2 (Schema 缓存) |
| **严重程度** | **Critical** |
| **类别** | 逻辑矛盾 |
| **描述** | FR-2 验收标准要求 Schema Token 总数不超过 4000 tokens，但 §4.1 性能要求中 100 表的 Schema 全量加载时间 < 3s。如果按 4000 tokens 上限，约相当于 3000 个英文单词或 2000 个中文字符的容量，对于一个 100 表的数据库，仅表名列表就可能超过此限制。4000 tokens 对于描述完整 Schema 来说是极小的预算。 |
| **建议** | 重新评估 Token 预算。对于 100 表的数据库，完整 Schema 可能需要 10k-50k tokens。建议：1) 将默认阈值改为实际测量的合理值；2) 区分"完整 Schema 缓存"和"送给 LLM 的 Schema Context"两个概念——缓存可以全量，Prompt 才需要精简。 |

### C2. 无状态架构与 Schema 缓存生命周期的矛盾

| 字段 | 内容 |
|------|------|
| **位置** | §5.3 架构约束 + §2.1 FR-2 |
| **严重程度** | **Critical** |
| **类别** | 架构矛盾 |
| **描述** | §5.3 声明 "无状态：MCP Server 不存储用户状态"，但 FR-2 要求缓存 Schema 且 "缓存随 Server 生命周期"。缓存本身就是一种状态。虽然不冲突（Schema 缓存是 Server 状态而非用户状态），但表述不严谨，可能导致后续开发者误解缓存刷新策略。 |
| **建议** | 将 "无状态" 改为 "无用户状态 (Stateless w.r.t. user sessions)"，明确 Schema 缓存属于 Server 的只读基础设施状态，不因用户不同而变化。 |

## 三、High 问题

### H1. 结果意义性验证（Layer 4）缺少样本量定义

| 字段 | 内容 |
|------|------|
| **位置** | §8.2 Execute 模式流程 + §2.1 FR-6 |
| **严重程度** | **High** |
| **类别** | 设计遗漏 |
| **描述** | §8.2 流程图中提到 "LLM 采样评估前 N 行"，但 FR-6 中未定义 N 的值。如果结果集有 100 万行，采样 1000 行和采样 10 行的成本和质量差异巨大。 |
| **建议** | 在 FR-6 中明确采样策略：固定行数（如 20 行）、比例采样（如 1%）、或自适应采样（结果集 < 100 行全量，> 100 行取前 100 行）。 |

### H2. 巨量结果集处理策略缺失

| 字段 | 内容 |
|------|------|
| **位置** | §2.1 FR-7 / FR-8 |
| **严重程度** | **High** |
| **类别** | 边界情况 |
| **描述** | FR-8 定义了分页但最大 page_size 为 10000。如果用户的查询没有 WHERE 条件返回数百万行，默认 page=1, page_size=100 时仍需从数据库拉取 100 行通过 JDBC 到内存。未说明结果集大小限制或流式处理的策略。 |
| **建议** | 1) 增加全局结果集行数上限（如 10000 行），超出时截断并提示；2) 考虑设置结果集内存上限（如 50MB），通过 JDBC Fetch Size 控制；3) 在 meta 中增加 truncated: boolean 字段标识是否被截断。 |

### H3. 并发场景下的 Schema 缓存一致性问题

| 字段 | 内容 |
|------|------|
| **位置** | §2.1 FR-2 + §4.1 |
| **严重程度** | **High** |
| **类别** | 并发安全 |
| **描述** | §4.1 要求支持 >= 10 并发查询。如果在缓存加载过程中（< 3s）有查询请求到达，是否需要阻塞等待？如果使用懒加载，并发请求会导致缓存击穿。 |
| **建议** | 明确缓存加载策略：启动时同步阻塞直到加载完成（简单）、或双缓冲（加载期间用旧缓存服务，新缓存就绪后原子切换）、或懒加载 + 锁机制。推荐启动时同步加载 + 健康检查确认就绪后再接受请求。 |

### H4. 只读账号权限粒度定义模糊

| 字段 | 内容 |
|------|------|
| **位置** | §4.2 安全需求 |
| **严重程度** | **High** |
| **类别** | 安全设计 |
| **描述** | "MCP Server 使用只读数据库账号"过于笼统。只读账号应限制到具体粒度：仅对特定数据库授予 SELECT 权限？是否需要 VIEW DEFINITION 权限来加载 Schema？是否需要 `sys.dm_exec_describe_first_result_set` 权限用于 Layer 3 验证？不同粒度影响安全性和功能完整性。 |
| **建议** | 明确所需的最低权限集：建议 `db_datareader` 角色成员 + `VIEW DEFINITION` 权限。列出需要的系统视图/函数访问权限。 |

### H5. LLM API Key 安全存储未具体化

| 字段 | 内容 |
|------|------|
| **位置** | §4.2 安全需求 |
| **严重程度** | **High** |
| **类别** | 安全遗漏 |
| **描述** | "密码、API Key 等使用安全存储方式"过于泛泛。Java 项目中常见的做法包括环境变量、Vault、Kubernetes Secret、Jasypt 加密配置文件等，不同方案风险等级和运维复杂度迥异。 |
| **建议** | 至少明确：1) 推荐环境变量（最低安全要求）；2) 可选集成 HashiCorp Vault（生产环境）；3) 配置文件中的敏感字段必须加密。 |

## 四、Medium 问题

### M1. `xp_cmdshell` 等存储过程只能通过 `EXEC` 调用—列表中冗余

| 字段 | 内容 |
|------|------|
| **位置** | §6.2 危险对象列表 |
| **严重程度** | **Medium** |
| **类别** | 逻辑冗余 |
| **描述** | 危险对象列表中同时列出了 `EXEC/EXECUTE` 和 `xp_cmdshell`、`sp_executesql` 等存储过程。由于 Layer 2 只允许 SELECT/WITH 语句，任何以 EXEC 开头的语句都会被拒绝，因此单独列出 xp_cmdshell 等作为禁止对象是冗余的。不过作为文档的防御纵深展示和审计依据，可以保留。 |
| **建议** | 添加注释说明：`EXEC/EXECUTE` 在 Layer 2 AST 层面拦截，`xp_cmdshell` 等列在 Layer 1 正则预检中作为辅助防御。分清楚哪些是 Layer 1 的规则、哪些是 Layer 2 的规则。 |

### M2. Schema 精简策略中关键词匹配的可靠性未评估

| 字段 | 内容 |
|------|------|
| **位置** | §2.1 FR-2 Schema 精简策略 |
| **严重程度** | **Medium** |
| **类别** | 设计风险 |
| **描述** | "基于表名和列名关键词匹配"来选择相关表是一个不精确的方法。例如查询 "用户订单总额" 需要 JOIN `users` 和 `orders` 表，但如果用户查询 "最近消费情况"，关键词匹配可能只命中 `orders` 表而遗漏 `users` 表，导致生成的 JOIN SQL 不完整。 |
| **建议** | 1) 利用外键关系扩展选中表（如果 orders 引用了 users 则自动包含 users）；2) 如果匹配的表少于 N 个，自动包含所有与该表有外键关系的相邻表；3) Schema 精简策略标记为 v2 可优化项。 |

### M3. LLM 重试策略未反映在错误码或流程中

| 字段 | 内容 |
|------|------|
| **位置** | §10 Q7 + §7.1 + §8 |
| **严重程度** | **Medium** |
| **类别** | 一致性问题 |
| **描述** | §10 Q7 记录了重试策略（最多 3 次，指数退避），但 §7.1 错误码表和 §8 处理流程中均未反映重试逻辑。如果第 1 次失败且重试后成功，返回正常响应；如果 3 次全部失败，返回 `LLM_API_ERROR`。流程图中没有这个分支。 |
| **建议** | 在 §8 的流程图中增加 "重试计数器 ≥ 3?" 判断分支，并在 §7.2 中增加重试信息的 meta 字段。 |

### M4. `page` 参数从 1 开始但 SQL 的 OFFSET 从 0 开始

| 字段 | 内容 |
|------|------|
| **位置** | §2.1 FR-8 / FR-9 |
| **严重程度** | **Medium** |
| **类别** | 实现注意事项 |
| **描述** | FR-9 定义 page 从 1 开始，但 SQL Server 的 `OFFSET ... ROWS FETCH NEXT ... ROWS ONLY` 中 OFFSET 从 0 开始。这是一个容易被忽略的实现细节，建议在文档中显式标注 OFFSET 的计算公式：`OFFSET = (page - 1) * page_size`。 |
| **建议** | 在 FR-8 或 FR-9 中添加一行计算公式说明。 |

### M5. 表注释/列注释的获取方式脆弱

| 字段 | 内容 |
|------|------|
| **位置** | §3.3 Schema 加载 SQL |
| **严重程度** | **Medium** |
| **类别** | 维护性 |
| **描述** | 使用 `sys.extended_properties` 的 `MS_Description` 获取注释是 SQL Server 的常规做法，但这个 JOIN 涉及 `OBJECT_ID()` 和 `COLUMNPROPERTY()` 函数，对于大数据库查询性能可能不理想。另外，对于视图列的注释，当前 SQL 未覆盖。 |
| **建议** | 考虑将注释加载移到后台异步任务，不阻塞主 Schema 加载流程。视图注释可以标记为 v2 待办。 |

### L1. `output_format` 参数在 `sql_only` 模式下无意义

|------|------|

| **严重程度** | **Low** |
