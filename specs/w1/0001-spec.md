## 1. 项目概述

### 1.1 背景与目标

构建一个轻量级的 **Ticket（工单/任务）管理工具**，通过 **标签（Tag）** 对 Ticket 进行分类与筛选，便于个人或小团队跟踪待办与已完成事项。

**本期不包含用户账号与权限体系**：假定单一使用场景，所有操作在「匿名/单用户」前提下进行。

### 1.2 技术约束（已定）

| 层次 | 技术 |
|------|------|
| 数据库 | Microsoft SQL Server |
| 后端 | Java + Spring Boot + Hibernate（JPA） |
| API 文档 | SpringDoc OpenAPI 3 + Swagger UI（浏览器内调试） |
| 前端 | JavaScript + Bootstrap |

### 1.2.1 技术栈版本说明

下表为 **建议基线版本**，用于第一周课程对齐环境；开题前可在 [Spring Boot](https://spring.io/projects/spring-boot)、[Adoptium / Oracle JDK](https://adoptium.net/)、[Bootstrap](https://getbootstrap.com/) 官网核对当前稳定版，并在团队中统一锁定补丁号。

## 3. 功能需求

### 3.1 Ticket 管理

| ID | 需求 | 验收要点 |
|----|------|----------|
| FR-T-01 | 创建 Ticket | 必填校验通过后持久化；列表或详情可见 |
| FR-T-02 | 编辑 Ticket | 标题、描述等可修改；保存后一致 |
| FR-T-03 | 删除 Ticket | 建议二次确认；关联标签关系一并清理 |
| FR-T-04 | 标记完成 | `completed` 为真；列表与详情一致 |
| FR-T-05 | 取消完成 | `completed` 恢复为假 |

**Ticket 最小字段：**

- `title`（必填）、`description`（可选）、`completed`（默认 false）
- `createdAt`、`updatedAt`（系统维护）

### 3.2 标签管理

| ID | 需求 | 验收要点 |
|----|------|----------|
| FR-G-01 | 为 Ticket 添加标签 | 同一 Ticket 上不重复同一标签 |
| FR-G-02 | 从 Ticket 移除标签 | 仅解除关联；是否删除「孤儿」Tag 由策略决定（见 5.2） |
| FR-G-03 | 标签实体可维护 | 支持独立创建标签，和/或在添加时按名称隐式创建 |

### 3.3 列表、筛选与搜索

| ID | 需求 | 验收要点 |
|----|------|----------|
| FR-L-01 | 展示 Ticket 列表 | 建议：未完成优先，再按 `updatedAt` 倒序 |
| FR-L-02 | 按标签查看列表 | 选一个或多个标签时列表随之过滤；AND/OR 规则见 4.2 |
| FR-L-03 | 按 title 搜索 | 标题模糊匹配；可与标签筛选同时生效 |

### 3.4 非功能需求

| ID | 说明 |
|----|------|
| NFR-01 | 界面使用 Bootstrap，主要操作有成功/失败反馈 |
| NFR-02 | 单用户场景下响应可接受；小数据量可先不分页 |
| NFR-03 | 删除 Ticket 时级联清理 `ticket_tag`，无孤儿关联 |
| NFR-04 | 前后端分离，REST API 契约清晰 |

## 4. 系统设计

### 4.1 架构

```
浏览器 (Bootstrap + JavaScript)
        │  HTTP/JSON (REST)
        ▼
Spring Boot（Controller → Service → Repository / JPA）
        │
        ▼
SQL Server
```

静态资源可由 Spring Boot `src/main/resources/static` 托管（工程根目录见第 1.2.2 节）；前后端端口分离时需配置 CORS。

### 4.2 多标签筛选策略（需实现时选定并在 UI 说明）

- **AND（推荐）**：选中 A、B 时，仅显示同时具有 A 与 B 的 Ticket。
- **OR（更简单）**：显示具有任意选中标签的 Ticket。

### 4.3 搜索与筛选组合

标签筛选与标题关键字搜索 **同时生效**（先按标签缩小集合再匹配标题，或等价顺序，结果一致即可）。

## 5. 数据模型与数据库设计

### 5.1 概念模型

- **Ticket**、**Tag** 为多对多，中间表 **ticket_tag**。
- 标签名在系统内 **唯一**（推荐）。

### 5.2 表结构

**`ticket`**

| 列 | 类型 | 约束 |
|----|------|------|
| id | BIGINT IDENTITY | PK |
| title | NVARCHAR(255) | NOT NULL |
| description | NVARCHAR(MAX) | NULL |
| completed | BIT | NOT NULL, DEFAULT 0 |
| created_at | DATETIME2 | NOT NULL |
| updated_at | DATETIME2 | NOT NULL |

**`tag`**

| 列 | 类型 | 约束 |
|----|------|------|
| id | BIGINT IDENTITY | PK |
| name | NVARCHAR(64) | NOT NULL, UNIQUE |

**`ticket_tag`**

| 列 | 类型 | 约束 |
|----|------|------|
| ticket_id | BIGINT | FK → ticket.id，复合 PK |
| tag_id | BIGINT | FK → tag.id，复合 PK |

删除 Ticket 时级联删除关联行。删除 Tag 可采用：仅无关联时可删（应用层 + `NO ACTION`），或删除 Tag 时级联删除关联行（`CASCADE`）；下文 DDL 采用 Tag 侧 **CASCADE** 解除关联，便于课程演示。

### 5.3 DDL 脚本（SQL Server）

执行前按需修改库名或使用 `USE`。

```sql
/* 可选：
CREATE DATABASE project_alpha;
GO
USE project_alpha;
GO
*/

CREATE TABLE ticket (
    id BIGINT IDENTITY(1, 1) NOT NULL,
    title NVARCHAR(255) NOT NULL,
    description NVARCHAR(MAX) NULL,
    completed BIT NOT NULL
        CONSTRAINT DF_ticket_completed DEFAULT (0),
    created_at DATETIME2(3) NOT NULL
        CONSTRAINT DF_ticket_created_at DEFAULT (SYSUTCDATETIME()),
    updated_at DATETIME2(3) NOT NULL
        CONSTRAINT DF_ticket_updated_at DEFAULT (SYSUTCDATETIME()),
    CONSTRAINT PK_ticket PRIMARY KEY (id)
);

CREATE TABLE tag (
    id BIGINT IDENTITY(1, 1) NOT NULL,
    name NVARCHAR(64) NOT NULL,
    CONSTRAINT PK_tag PRIMARY KEY (id),
    CONSTRAINT UQ_tag_name UNIQUE (name)
);

CREATE TABLE ticket_tag (
    ticket_id BIGINT NOT NULL,
    tag_id BIGINT NOT NULL,
    CONSTRAINT PK_ticket_tag PRIMARY KEY (ticket_id, tag_id),
    CONSTRAINT FK_ticket_tag_ticket
        FOREIGN KEY (ticket_id) REFERENCES ticket (id) ON DELETE CASCADE,
    CONSTRAINT FK_ticket_tag_tag
        FOREIGN KEY (tag_id) REFERENCES tag (id) ON DELETE CASCADE
);

CREATE NONCLUSTERED INDEX IX_ticket_title ON ticket (title);
CREATE NONCLUSTERED INDEX IX_ticket_tag_tag_id ON ticket_tag (tag_id);
```

### 5.4 Hibernate / JPA 要点

- `Ticket` 与 `Tag`：`@ManyToMany`，`@JoinTable(name = "ticket_tag", ...)`，注意拥有方与 `cascade`，避免误删全局 Tag。
- 时间字段：`@CreationTimestamp` / `@UpdateTimestamp` 或由 Service 统一赋值。
- `completed` 映射为 `boolean` ↔ `BIT`。

## 6. API 设计（REST）

统一前缀：`/api`。JSON 使用 **camelCase**。错误时返回合适 HTTP 状态码与 JSON 错误体（可与 Spring `ProblemDetail` 或自定义结构对齐）。

### 6.1 Ticket

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/tickets` | 列表；查询参数：`tagIds`（多值）、`q`（标题关键字） |
| GET | `/api/tickets/{id}` | 详情（含 tags） |
| POST | `/api/tickets` | 创建 |
| PUT | `/api/tickets/{id}` | 更新（全量或约定局部） |
| DELETE | `/api/tickets/{id}` | 删除 |
| PATCH | `/api/tickets/{id}/complete` | 标记完成 |
| PATCH | `/api/tickets/{id}/incomplete` | 取消完成 |

*可选：合并为 `PATCH /api/tickets/{id}`，Body `{ "completed": true/false }`。*

### 6.2 Tag 与关联

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/tags` | 全部标签 |
| POST | `/api/tags` | 新建标签 |
| POST | `/api/tickets/{id}/tags` | 添加关联（`tagId` 或 `tagName`） |
| DELETE | `/api/tickets/{id}/tags/{tagId}` | 移除关联 |

### 6.3 DTO 与校验

- Bean Validation：`title` 非空、长度等。
- `tagIds`、`q` 缺省表示不过滤该项。

假定 `http://localhost:8080`，时间为 ISO-8601 风格字符串。

### 6.5 API 文档（Swagger / OpenAPI）

后端使用 **SpringDoc OpenAPI**（`springdoc-openapi-starter-webmvc-ui`）暴露 **OpenAPI 3** 描述与 **Swagger UI**。日常开发中在浏览器打开 UI 即可查看路径、参数、模型，并直接 **Try it out** 发请求，无需单独维护一份与代码脱节的文档。

**与 Swagger 的关系**：业界常说的「Swagger」现多指 **OpenAPI 规范** + **Swagger UI** 这套组合；Spring Boot 3 推荐 SpringDoc，而非已较少维护的 Springfox。

**Maven 依赖示例**

```xml
<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
    <version>2.8.4</version>
</dependency>
```

*版本号实现时可改为当前稳定版。*

**Gradle（Kotlin DSL）示例**

```kotlin
implementation("org.springdoc:springdoc-openapi-starter-webmvc-ui:2.8.4")
```

**启动后访问地址（默认，未改配置时）**

| 用途 | 路径（相对服务根） |
|------|---------------------|
| Swagger UI | `/swagger-ui/index.html`（部分环境也可从 `/swagger-ui.html` 重定向） |
| OpenAPI JSON | `/v3/api-docs` |
| OpenAPI YAML | `/v3/api-docs.yaml` |

**`application.yml` 可选配置示例**

```yaml
springdoc:
  api-docs:
    path: /v3/api-docs
  swagger-ui:
    path: /swagger-ui
    operations-sorter: method
    tags-sorter: alpha
```

**代码侧建议（与 UI 展示质量相关）**

- 启动类或配置类上使用 `@OpenAPIDefinition`，填写 `info`（标题、描述、版本）。  
- Controller 上使用 `@Tag`；接口上使用 `@Operation`、`@ApiResponses`。  
- DTO 上使用 `@Schema` 描述字段说明、必填、示例值。  
- 列表查询的 `tagIds` 多值参数用 `@Parameter` 说明重复键或逗号分隔等约定。  

**本期无登录**：一般不在 OpenAPI 中配置 `securitySchemes`；若后续加 JWT，再在 SpringDoc 中补充全局 `SecurityRequirement`。

**契约文件**：仓库内提供与本文一致的 OpenAPI 3 描述文件 `specs/w1/openapi.yaml`，可用于导入 Swagger Editor、Postman，或与实现中的 `/v3/api-docs` 对照；**以运行中应用生成的文档为准**时，需在变更接口后同步更新该 YAML（或改为由构建从 `api-docs` 导出）。
