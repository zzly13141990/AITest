## [E] Error（必须修复）

### E1. 查询结果过大 HTTP 状态码与设计方案不一致

**文件位置**: 第 10.1 节（错误码表）  
**严重性**: [E] Error  
**问题描述**:

第 10.1 节错误码表中，"查询结果过大" 使用 HTTP 状态码 **413**。但技术设计方案（docs/0001-view-to-api-design.md v1.4 section 2.7）已将同一场景定义为 **422 Unprocessable Entity**（设计 v1.1 审查后整改项）。

RFC 定义：413 表示请求实体过大（Request Entity Too Large），而当前场景是响应实体过大，应使用 422 更合理。

**建议修复**:

将 PRD 第 10.1 节的 `413` 改为 `422`，保持与设计文档一致。

## [W] Warning（建议修复）

### W1. 管理端 API 安全缺少部署层防护说明

**文件位置**: 第 12 节（部署建议）  
**严重性**: [W] Warning  
**问题描述**:

PRD 声明"不提供管理端登录认证（管理页面为内网运维工具）"，但未在部署章节中说明如何通过基础设施层面的访问控制来保护管理端 API。前端管理页面和 API 部署在同一 JAR 中，任何知道 URL 的人都可以访问日志（含完整 SQL）。

**建议修复**:

在 12.2 节或新增 12.5 节中补充管理端 API 的访问控制说明，至少包括：
- 通过 Nginx / API 网关对 /api/v1/admin/* 配置内网 IP 白名单。
- 或通过 Spring Security 的 IP 过滤器实现基本防护。
- 明确这些措施属于部署运维层面，不纳入应用认证范围（符合 PRD 范围定义）。

## [I] Info（建议考虑）

### I1. 响应时间字段缺少时区说明

**文件位置**: 第 5.3 节（响应格式）  
**严重性**: [I] Info  
**问题描述**:

`execution_time` 字段格式为 `yyyy-MM-dd HH:mm:ss`，但没有指定时区。当客户端和服务端不在同一时区时可能导致时间理解偏差。

**建议**: 在字段说明中注明时区，如"服务端本地时区"或统一使用 UTC。

### I2. 非分页查询上限 10000 条但未说明性能影响

**文件位置**: 第 6.2 节（分页实现策略）  
**严重性**: [I] Info  
**问题描述**:

文档提到"未传入 page 参数时：直接执行原始 SQL，上限 10000 条"。但未说明在未分页情况下查询 10000 条数据对目标数据库和网络传输的潜在影响。

**建议**: 增加说明，建议第三方调用时始终带分页参数，无分页查询仅用于小数据量确认场景。
